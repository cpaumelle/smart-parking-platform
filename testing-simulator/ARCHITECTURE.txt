## Data Flow Detail

### Uplink Flow (Sensor → Application)

```
[Parking Sensor]
      ↓
  Measures distance (ultrasonic)
      ↓
  Detects car present/absent
      ↓
  Constructs uplink payload:
  [Distance (2B)] [Battery (1B)] [State (1B)] [Counter (1B)]
      ↓
  Send to ChirpStack API
  POST /devices/{dev_eui}/queue
      ↓
[ChirpStack Server]
      ↓
  Processes uplink
      ↓
  Forwards to Integration/Webhook
      ↓
[Your Application]
```

### Downlink Flow (Application → Busylight)

```
[Your Application]
      ↓
  Determines required color based on parking state
      ↓
  Constructs downlink payload:
  [R (1B)] [G (1B)] [B (1B)] [On-time (1B)] [Off-time (1B)]
      ↓
  Enqueue to ChirpStack API
  POST /devices/{dev_eui}/queue
      ↓
[ChirpStack Server]
      ↓
  Queues downlink for device
      ↓
  Waits for next receive window (Class C = immediate)
      ↓
  Transmits downlink
      ↓
[Busylight]
      ↓
  Receives command
      ↓
  Sets LED color
  🟢 🔴 🟠 🔵 ⚪
```

## Message Format

### Sensor Uplink (5 bytes)
```
┌─────────┬─────────┬─────────┬─────────┬─────────┐
│ Byte 0-1│  Byte 2 │  Byte 3 │  Byte 4 │         │
│ Distance│ Battery │  State  │ Counter │         │
│  (cm)   │  (%)    │   enum  │  uint8  │         │
│ uint16  │  uint8  │  uint8  │         │         │
└─────────┴─────────┴─────────┴─────────┴─────────┘

States:
  0 = Available
  1 = Occupied
  2 = Reserved
  3 = Maintenance
  255 = Unknown
```

### Busylight Downlink (5 bytes)
```
┌─────────┬─────────┬─────────┬─────────┬─────────┐
│  Byte 0 │  Byte 1 │  Byte 2 │  Byte 3 │  Byte 4 │
│   Red   │  Green  │  Blue   │ On-time │Off-time │
│  0-255  │  0-255  │  0-255  │  0-255  │  0-255  │
└─────────┴─────────┴─────────┴─────────┴─────────┘

Examples:
  Green:  00 64 00 FF 00  (Solid)
  Red:    64 00 00 FF 00  (Solid)
  Orange: FF A5 00 FF 00  (Solid)
  Blink:  FF A5 00 7F 7F  (Blinking)
```

## Key Design Decisions

1. **1:1 Mapping**: Each sensor has exactly one busylight
2. **Class A Sensors**: Send uplinks periodically or on change
3. **Class C Busylights**: Always listening for downlinks
4. **Realistic Timing**: Configurable update intervals with jitter
5. **Statistical Behavior**: Random parking durations with normal distribution
6. **Scenario Support**: Time-of-day multipliers for realistic patterns
7. **Mock Mode**: Testing without infrastructure
8. **Thread Safety**: Single simulation thread for simplicity

## Performance Characteristics

- **100 devices** = manageable load for single ChirpStack instance
- **Update interval**: 60 seconds default (configurable)
- **Message rate**: ~1.67 msg/sec average, ~16 msg/sec peak
- **Memory**: ~50MB for 100 devices + simulator overhead
- **CPU**: Minimal (<5% on modern CPU)
- **Network**: ~1 KB/sec bandwidth average

---

This architecture provides a realistic, scalable simulation platform
for testing smart parking systems with 100+ devices simultaneously.
